50 /*******************************************************************************
51 Replication SQL Thread
52 
53 Relay_log_info contains:
54  - the current relay log
55  - the current relay log offset
56  - master log name
57  - master log sequence corresponding to the last update
58  - misc information specific to the SQL thread
59 
60 Relay_log_info is initialized from a repository, i.e. table or file, if there is
61 one. Otherwise, data members are intialized with defaults by calling
62 init_relay_log_info().
63 
64 The relay.info table/file shall be updated whenever: (i) the relay log file
65 is rotated, (ii) SQL Thread is stopped, (iii) while processing a Xid_log_event,
66 (iv) after a Query_log_event (i.e. commit or rollback) and (v) after processing
67 any statement written to the binary log without a transaction context.
68 
69 The Xid_log_event is a commit for transactional engines and must be handled
70 differently to provide reliability/data integrity. In this case, positions
71 are updated within the context of the current transaction. So
72 
73  . If the relay.info is stored in a transactional repository and the server
74  crashes before successfully committing the transaction the changes to the
75  position table will be rolled back along with the data.
76 
77  . If the relay.info is stored in a non-transactional repository, for instance,
78  a file or a system table created using MyIsam, and the server crashes before
79  successfully committing the transaction the changes to the position table
80  will not be rolled back but data will.
81 
82 In particular, when there are mixed transactions, i.e a transaction that updates
83 both transaction and non-transactional engines, the Xid_log_event is still used
84 but reliability/data integrity cannot be achieved as we shall explain in what
85 follows.
86 
87 Changes to non-transactional engines, such as MyIsam, cannot be rolled back if a
88 failure happens. For that reason, there is no point in updating the positions
89 within the boundaries of any on-going transaction. This is true for both commit
90 and rollback. If a failure happens after processing the pseudo-transaction but
91 before updating the positions, the transaction will be re-executed when the
92 slave is up most likely causing an error that needs to be manually circumvented.
93 This is a well-known issue when non-transactional statements are executed.
94 
95 Specifically, if rolling back any transaction, positions are updated outside the
96 transaction boundaries. However, there may be a problem in this scenario even
97 when only transactional engines are updated. This happens because if there is a
98 rollback and such transaction is written to the binary log, a non-transactional
99 engine was updated or a temporary table was created or dropped within its
100 boundaries.
101 
102 In particular, in both STATEMENT and MIXED logging formats, this happens because
103 any temporary table is automatically dropped after a shutdown/startup.
104 See BUG#26945 for further details.
105 
106 Statements written to the binary log outside the boundaries of a transaction are
107 DDLs or maintenance commands which are not transactional. These means that they
108 cannot be rolled back if a failure happens. In such cases, the positions are
109 updated after processing the events. If a failure happens after processing the
110 statement but before updating the positions, the statement will be
111 re-executed when the slave is up most likely causing an error that needs to be
112 manually circumvented. This is a well-known issue when non-transactional
113 statements are executed.
114 
115 The --sync-relay-log-info does not have effect when a system table, either
116 transactional or non-transactional is used.
117 
118 To correctly recovery from failures, one should combine transactional system
119 tables along with the --relay-log-recovery.
120 *******************************************************************************/
