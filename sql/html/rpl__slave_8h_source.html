82 /*****************************************************************************
83 
84  MySQL Replication
85 
86  Replication is implemented via two types of threads:
87 
88  I/O Thread - One of these threads is started for each master server.
89  They maintain a connection to their master server, read log
90  events from the master as they arrive, and queues them into
91  a single, shared relay log file. A Master_info
92  represents each of these threads.
93 
94  SQL Thread - One of these threads is started and reads from the relay log
95  file, executing each event. A Relay_log_info
96  represents this thread.
97 
98  Buffering in the relay log file makes it unnecessary to reread events from
99  a master server across a slave restart. It also decouples the slave from
100  the master where long-running updates and event logging are concerned--ie
101  it can continue to log new events while a slow query executes on the slave.
102 
103 *****************************************************************************/
104 
105 /*
106  MUTEXES in replication:
107 
108  channel_map lock: This is to lock the Multisource datastructure (channel_map).
109  Generally it used to retrieve an mi from channel_map.It is used to SERIALIZE ALL
110  administrative commands of replication: START SLAVE, STOP SLAVE, CHANGE
111  MASTER, RESET SLAVE, end_slave() (when mysqld stops) [init_slave() does not
112  need it it's called early]. Any of these commands holds the mutex from the
113  start till the end. This thus protects us against a handful of deadlocks
114  (consider start_slave_thread() which, when starting the I/O thread, releases
115  mi->run_lock, keeps rli->run_lock, and tries to re-acquire mi->run_lock).
116 
117  In Master_info: run_lock, data_lock
118  run_lock protects all information about the run state: slave_running, thd
119  and the existence of the I/O thread (to stop/start it, you need this mutex).
120  data_lock protects some moving members of the struct: counters (log name,
121  position) and relay log (MYSQL_BIN_LOG object).
122 
123  In Relay_log_info: run_lock, data_lock
124  see Master_info
125  However, note that run_lock does not protect
126  Relay_log_info.run_state; that is protected by data_lock.
127 
128  In MYSQL_BIN_LOG: LOCK_log, LOCK_index of the binlog and the relay log
129  LOCK_log: when you write to it. LOCK_index: when you create/delete a binlog
130  (so that you have to update the .index file).
131 
132  The global_sid_lock must not be taken after LOCK_reset_gtid_table.
133 
134  ==== Order of acquisition ====
135 
136  Here, we list most major functions that acquire multiple locks.
137 
138  Notation: For each function, we list the locks it takes, in the
139  order it takes them. If a function holds lock A while taking lock
140  B, then we write "A, B". If a function locks A, unlocks A, then
141  locks B, then we write "A | B". If function F1 invokes function F2,
142  then we write F2's name in parentheses in the list of locks for F1.
143 
144  show_master_info:
145  mi.data_lock, rli.data_lock, mi.err_lock, rli.err_lock
146 
147  stop_slave:
148  channel_map lock,
149  ( mi.run_lock, thd.LOCK_thd_data
150  | rli.run_lock, thd.LOCK_thd_data
151  | relay.LOCK_log
152  )
153 
154  start_slave:
155  mi.run_lock, rli.run_lock, rli.data_lock, global_sid_lock->wrlock
156 
157  reset_logs:
158  THD::LOCK_thd_data, .LOCK_log, .LOCK_index, global_sid_lock->wrlock
159 
160  purge_relay_logs:
161  rli.data_lock, (relay.reset_logs) THD::LOCK_thd_data,
162  relay.LOCK_log, relay.LOCK_index, global_sid_lock->wrlock
163 
164  reset_master:
165  (binlog.reset_logs) THD::LOCK_thd_data, binlog.LOCK_log,
166  binlog.LOCK_index, global_sid_lock->wrlock, LOCK_reset_gtid_table
167 
168  reset_slave:
169  mi.run_lock, rli.run_lock, (purge_relay_logs) rli.data_lock,
170  THD::LOCK_thd_data, relay.LOCK_log, relay.LOCK_index,
171  global_sid_lock->wrlock
172 
173  purge_logs:
174  .LOCK_index, LOCK_thd_list, thd.linfo.lock
175 
176  [Note: purge_logs contains a known bug: LOCK_index should not be
177  taken before LOCK_thd_list. This implies that, e.g.,
178  purge_master_logs can deadlock with reset_master. However,
179  although purge_first_log and reset_slave take locks in reverse
180  order, they cannot deadlock because they both first acquire
181  rli.data_lock.]
182 
183  purge_master_logs, purge_master_logs_before_date, purge:
184  (binlog.purge_logs) binlog.LOCK_index, LOCK_thd_list, thd.linfo.lock
185 
186  purge_first_log:
187  rli.data_lock, relay.LOCK_index, rli.log_space_lock,
188  (relay.purge_logs) LOCK_thd_list, thd.linfo.lock
189 
190  MYSQL_BIN_LOG::new_file_impl:
191  .LOCK_log, .LOCK_index,
192  ( [ if binlog: LOCK_prep_xids ]
193  | global_sid_lock->wrlock
194  )
195 
196  rotate_relay_log:
197  (relay.new_file_impl) relay.LOCK_log, relay.LOCK_index,
198  global_sid_lock->wrlock
199 
200  kill_zombie_dump_threads:
201  LOCK_thd_list, thd.LOCK_thd_data
202 
203  init_relay_log_pos:
204  rli.data_lock, relay.log_lock
205 
206  rli_init_info:
207  rli.data_lock,
208  ( relay.log_lock
209  | global_sid_lock->wrlock
210  | (relay.open_binlog)
211  | (init_relay_log_pos) rli.data_lock, relay.log_lock
212  )
213 
214  change_master:
215  mi.run_lock, rli.run_lock, (init_relay_log_pos) rli.data_lock,
216  relay.log_lock
217 
218  Sys_var_gtid_mode::global_update:
219  gtid_mode_lock, channel_map lock, binlog.LOCK_log, global_sid_lock
220 
221  So the DAG of lock acquisition order (not counting the buggy
222  purge_logs) is, empirically:
223 
224  gtid_mode_lock, channel_map lock, mi.run_lock, rli.run_lock,
225  ( rli.data_lock,
226  ( LOCK_thd_list,
227  (
228  ( binlog.LOCK_log, binlog.LOCK_index
229  | relay.LOCK_log, relay.LOCK_index
230  ),
231  ( rli.log_space_lock | global_sid_lock->wrlock )
232  | binlog.LOCK_log, binlog.LOCK_index, LOCK_prep_xids
233  | thd.LOCK_data
234  )
235  | mi.err_lock, rli.err_lock
236  )
237  )
238  )
239  | mi.data_lock, rli.data_lock
240 */
