155 /*
156  Quick select interface.
157  This class is a parent for all QUICK_*_SELECT and FT_SELECT classes.
158 
159  The usage scenario is as follows:
160  1. Create quick select
161  quick= new QUICK_XXX_SELECT(...);
162 
163  2. Perform lightweight initialization. This can be done in 2 ways:
164  2.a: Regular initialization
165  if (quick->init())
166  {
167  //the only valid action after failed init() call is delete
168  delete quick;
169  }
170  2.b: Special initialization for quick selects merged by QUICK_ROR_*_SELECT
171  if (quick->init_ror_merged_scan())
172  delete quick;
173 
174  3. Perform zero, one, or more scans.
175  while (...)
176  {
177  // initialize quick select for scan. This may allocate
178  // buffers and/or prefetch rows.
179  if (quick->reset())
180  {
181  //the only valid action after failed reset() call is delete
182  delete quick;
183  //abort query
184  }
185 
186  // perform the scan
187  do
188  {
189  res= quick->get_next();
190  } while (res && ...)
191  }
192 
193  4. Delete the select:
194  delete quick;
195 
196  NOTE
197  quick select doesn't use Sql_alloc/MEM_ROOT allocation because "range
198  checked for each record" functionality may create/destroy
199  O(#records_in_some_table) quick selects during query execution.
200  See Bug#18684036 ELIMINATE LAST FEW HEAP USAGE FROM THE
201  RANGE OPTIMIZER.
202 */
203

533 /*
534  QUICK_INDEX_MERGE_SELECT - index_merge access method quick select.
535 
536  QUICK_INDEX_MERGE_SELECT uses
537  * QUICK_RANGE_SELECTs to get rows
538  * Unique class to remove duplicate rows
539 
540  INDEX MERGE OPTIMIZER
541  Current implementation doesn't detect all cases where index_merge could
542  be used, in particular:
543  * index_merge will never be used if range scan is possible (even if
544  range scan is more expensive)
545 
546  * index_merge+'using index' is not supported (this the consequence of
547  the above restriction)
548 
549  * If WHERE part contains complex nested AND and OR conditions, some ways
550  to retrieve rows using index_merge will not be considered. The choice
551  of read plan may depend on the order of conjuncts/disjuncts in WHERE
552  part of the query, see comments near imerge_list_or_list and
553  SEL_IMERGE::or_sel_tree_with_checks functions for details.
554 
555  * There is no "index_merge_ref" method (but index_merge on non-first
556  table in join is possible with 'range checked for each record').
557 
558  See comments around SEL_IMERGE class and test_quick_select for more
559  details.
560 
561  ROW RETRIEVAL ALGORITHM
562 
563  index_merge uses Unique class for duplicates removal. index_merge takes
564  advantage of Clustered Primary Key (CPK) if the table has one.
565  The index_merge algorithm consists of two phases:
566 
567  Phase 1 (implemented in QUICK_INDEX_MERGE_SELECT::prepare_unique):
568  prepare()
569  {
570  activate 'index only';
571  while(retrieve next row for non-CPK scan)
572  {
573  if (there is a CPK scan and row will be retrieved by it)
574  skip this row;
575  else
576  put its rowid into Unique;
577  }
578  deactivate 'index only';
579  }
580 
581  Phase 2 (implemented as sequence of QUICK_INDEX_MERGE_SELECT::get_next
582  calls):
583 
584  fetch()
585  {
586  retrieve all rows from row pointers stored in Unique;
587  free Unique;
588  retrieve all rows for CPK scan;
589  }
590 */
