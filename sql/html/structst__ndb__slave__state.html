◆ atTransactionCommit()

void st_ndb_slave_state::atTransactionCommit ( Uint64  epoch)

atTransactionCommit()

Called by Slave SQL thread after transaction commit

Conflict detected locally

Update last_conflicted_epoch if we applied reflected or refresh ops (Implies Secondary role in asymmetric algorithms)

This epoch which has looped the circle was stable - no new conflicts have been found / corrected since it was logged

Note that max_rep_epoch >= last_conflicted_epoch implies that there are no currently known-about conflicts. On the primary this is a definitive fact as it finds out about all conflicts immediately. On the secondary it does not mean that there are not committed conflicts, just that they have not started being corrected yet.
