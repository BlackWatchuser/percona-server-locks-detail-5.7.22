3519  /*
3520  Here is an outline of on-line/in-place ALTER TABLE execution through
3521  this interface.
3522 
3523  Phase 1 : Initialization
3524  ========================
3525  During this phase we determine which algorithm should be used
3526  for execution of ALTER TABLE and what level concurrency it will
3527  require.
3528 
3529  *) This phase starts by opening the table and preparing description
3530  of the new version of the table.
3531  *) Then we check if it is impossible even in theory to carry out
3532  this ALTER TABLE using the in-place algorithm. For example, because
3533  we need to change storage engine or the user has explicitly requested
3534  usage of the "copy" algorithm.
3535  *) If in-place ALTER TABLE is theoretically possible, we continue
3536  by compiling differences between old and new versions of the table
3537  in the form of HA_ALTER_FLAGS bitmap. We also build a few
3538  auxiliary structures describing requested changes and store
3539  all these data in the Alter_inplace_info object.
3540  *) Then the handler::check_if_supported_inplace_alter() method is called
3541  in order to find if the storage engine can carry out changes requested
3542  by this ALTER TABLE using the in-place algorithm. To determine this,
3543  the engine can rely on data in HA_ALTER_FLAGS/Alter_inplace_info
3544  passed to it as well as on its own checks. If the in-place algorithm
3545  can be used for this ALTER TABLE, the level of required concurrency for
3546  its execution is also returned.
3547  If any errors occur during the handler call, ALTER TABLE is aborted
3548  and no further handler functions are called.
3549  *) Locking requirements of the in-place algorithm are compared to any
3550  concurrency requirements specified by user. If there is a conflict
3551  between them, we either switch to the copy algorithm or emit an error.
3552 
3553  Phase 2 : Execution
3554  ===================
3555 
3556  In this phase the operations are executed.
3557 
3558  *) As the first step, we acquire a lock corresponding to the concurrency
3559  level which was returned by handler::check_if_supported_inplace_alter()
3560  and requested by the user. This lock is held for most of the
3561  duration of in-place ALTER (if HA_ALTER_INPLACE_SHARED_LOCK_AFTER_PREPARE
3562  or HA_ALTER_INPLACE_NO_LOCK_AFTER_PREPARE were returned we acquire an
3563  exclusive lock for duration of the next step only).
3564  *) After that we call handler::ha_prepare_inplace_alter_table() to give the
3565  storage engine a chance to update its internal structures with a higher
3566  lock level than the one that will be used for the main step of algorithm.
3567  After that we downgrade the lock if it is necessary.
3568  *) After that, the main step of this phase and algorithm is executed.
3569  We call the handler::ha_inplace_alter_table() method, which carries out the
3570  changes requested by ALTER TABLE but does not makes them visible to other
3571  connections yet.
3572  *) We ensure that no other connection uses the table by upgrading our
3573  lock on it to exclusive.
3574  *) a) If the previous step succeeds, handler::ha_commit_inplace_alter_table() is
3575  called to allow the storage engine to do any final updates to its structures,
3576  to make all earlier changes durable and visible to other connections.
3577  b) If we have failed to upgrade lock or any errors have occured during the
3578  handler functions calls (including commit), we call
3579  handler::ha_commit_inplace_alter_table()
3580  to rollback all changes which were done during previous steps.
3581 
3582  Phase 3 : Final
3583  ===============
3584 
3585  In this phase we:
3586 
3587  *) Update SQL-layer data-dictionary by installing .FRM file for the new version
3588  of the table.
3589  *) Inform the storage engine about this change by calling the
3590  handler::ha_notify_table_changed() method.
3591  *) Destroy the Alter_inplace_info and handler_ctx objects.
3592 
3593  */
