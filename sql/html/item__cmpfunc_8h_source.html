544 /*
545  trigcond<param>(arg) ::= param? arg : TRUE
546 
547  The class Item_func_trig_cond is used for guarded predicates
548  which are employed only for internal purposes.
549  A guarded predicate is an object consisting of an a regular or
550  a guarded predicate P and a pointer to a boolean guard variable g.
551  A guarded predicate P/g is evaluated to true if the value of the
552  guard g is false, otherwise it is evaluated to the same value that
553  the predicate P: val(P/g)= g ? val(P):true.
554  Guarded predicates allow us to include predicates into a conjunction
555  conditionally. Currently they are utilized for pushed down predicates
556  in queries with outer join operations.
557 
558  In the future, probably, it makes sense to extend this class to
559  the objects consisting of three elements: a predicate P, a pointer
560  to a variable g and a firing value s with following evaluation
561  rule: val(P/g,s)= g==s? val(P) : true. It will allow us to build only
562  one item for the objects of the form P/g1/g2...
563 
564  Objects of this class are built only for query execution after
565  the execution plan has been already selected. That's why this
566  class needs only val_int out of generic methods.
567 
568  Current uses of Item_func_trig_cond objects:
569  - To wrap selection conditions when executing outer joins
570  - To wrap condition that is pushed down into subquery
571 */

2078 /*
2079  The class Item_equal is used to represent conjunctions of equality
2080  predicates of the form field1 = field2, and field=const in where
2081  conditions and on expressions.
2082 
2083  All equality predicates of the form field1=field2 contained in a
2084  conjunction are substituted for a sequence of items of this class.
2085  An item of this class Item_equal(f1,f2,...fk) represents a
2086  multiple equality f1=f2=...=fk.
2087 
2088  If a conjunction contains predicates f1=f2 and f2=f3, a new item of
2089  this class is created Item_equal(f1,f2,f3) representing the multiple
2090  equality f1=f2=f3 that substitutes the above equality predicates in
2091  the conjunction.
2092  A conjunction of the predicates f2=f1 and f3=f1 and f3=f2 will be
2093  substituted for the item representing the same multiple equality
2094  f1=f2=f3.
2095  An item Item_equal(f1,f2) can appear instead of a conjunction of
2096  f2=f1 and f1=f2, or instead of just the predicate f1=f2.
2097 
2098  An item of the class Item_equal inherits equalities from outer
2099  conjunctive levels.
2100 
2101  Suppose we have a where condition of the following form:
2102  WHERE f1=f2 AND f3=f4 AND f3=f5 AND ... AND (...OR (f1=f3 AND ...)).
2103  In this case:
2104  f1=f2 will be substituted for Item_equal(f1,f2);
2105  f3=f4 and f3=f5 will be substituted for Item_equal(f3,f4,f5);
2106  f1=f3 will be substituted for Item_equal(f1,f2,f3,f4,f5);
2107 
2108  An object of the class Item_equal can contain an optional constant
2109  item c. Then it represents a multiple equality of the form
2110  c=f1=...=fk.
2111 
2112  Objects of the class Item_equal are used for the following:
2113 
2114  1. An object Item_equal(t1.f1,...,tk.fk) allows us to consider any
2115  pair of tables ti and tj as joined by an equi-condition.
2116  Thus it provide us with additional access paths from table to table.
2117 
2118  2. An object Item_equal(t1.f1,...,tk.fk) is applied to deduce new
2119  SARGable predicates:
2120  f1=...=fk AND P(fi) => f1=...=fk AND P(fi) AND P(fj).
2121  It also can give us additional index scans and can allow us to
2122  improve selectivity estimates.
2123 
2124  3. An object Item_equal(t1.f1,...,tk.fk) is used to optimize the
2125  selected execution plan for the query: if table ti is accessed
2126  before the table tj then in any predicate P in the where condition
2127  the occurrence of tj.fj is substituted for ti.fi. This can allow
2128  an evaluation of the predicate at an earlier step.
2129 
2130  When feature 1 is supported they say that join transitive closure
2131  is employed.
2132  When feature 2 is supported they say that search argument transitive
2133  closure is employed.
2134  Both features are usually supported by preprocessing original query and
2135  adding additional predicates.
2136  We do not just add predicates, we rather dynamically replace some
2137  predicates that can not be used to access tables in the investigated
2138  plan for those, obtained by substitution of some fields for equal fields,
2139  that can be used.
2140

2142  Item_equal are created only at the time a PS/SP is executed and
